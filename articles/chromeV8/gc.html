垃圾回收 #

为什么需要垃圾回收 #

在 V8 执行代码的过程中，当遇到函数时，会创建执行上下文并添加到栈的顶端，执行上下文中包含的函数声明的所有变量，当函数执行完毕后，对应的执行上下文从栈底弹出，函数的作用域随之销毁，包含的变量一会统一释放并被自动回收。

如果这些变量不被回收，持续占用内存，那么就会导致内存暴增。

在其他语言例如 C 和 C++中，都需要手动申请和释放内存，而例如 java 以及 js，底层引擎已经帮我们自动进行了内存的分配和管理，从而让我们集中精力写业务逻辑。然而因为一些代码不够严谨，而常常会引发内存泄漏。

V8 的内存限制 #

V8 引擎中的内存使用并非无限制的，在 64 位系统下最多使用 1.4GB 左右，而在 32 位系统下则只能使用一半，也就是 0.7GB 左右。虽然我们能够手动设置 Node 的内存上限，但是很少会遇到浏览器端使用大量内存的场景。最主要的是：垃圾回收本身也是一个耗时操作，如果内存使用过多，必然导致垃圾回收时间变长，影响主线程的运行。

V8 的垃圾回收策略 #

V8 的垃圾回收策略基于对象的存活时间，区分新生代和老生代，采取不同的垃圾回收算法。

V8 的内存结构 #

V8 的内存结构主要包含如下几个部分：

  • 新生代：区域小，垃圾回收频繁，大多数的对象开始都会被分配在这里。此区域还分为两部分，From和To空间，以便于 GC 复制。

  • 老生代：区域大，新生代的对象存活久就会被转移到这里，垃圾回收频率较低。

  • 大对象区：存放体积比其他区域都大的对象，每个对象都会有自己的内存，垃圾回收不会移动这里。

  • 代码区：存放 jit 编译的代码。

  • map-space：存放隐藏类的地方。

新生代 #

新生代由 From 和 To 空间等分构成；主要采用Scavenge算法实现垃圾回收。

原因在于：

新生代的对象存活时间较短，空间占用不会很大，gc 执行的频率高。这个算法就是用空间来换时间，保证执行的时间尽可能短；

算法的流程如下：

  1. 我们声明的对象首先被分配到 From 空间，假设我们在 From 空间中分配了三个对象 A、B、C

  2. 当主线程第一次执行完毕后进入垃圾回收时，A 对象没有其他引用，也就可以回收。

  3. B、C 对象仍旧活跃，被复制到 To 空间进行保存。

  4. From 空间全部清空。

  5. From 空间和 To 空间互换

  6. 循环往复

过程还是很简单的， 这个算法的缺点就在于要浪费一般的内存用于复制

对象晋升 #

当一个对象在经过多次复制后依旧存活，那么再下一次垃圾回收时，就会被转移到老生代中，称为对象晋升 需要满足以下条件：

  • 对象是否经历过一次 Scavenge 算法
  • To 空间的内存占比是否已经超过 25%

在对象从 From 空间复制到 To 空间之前，会检查该对象是否经历过一次 Scavenge 算法，如果对象的地址已经发生了变化，则不会复制到 To 空间，而是直接转移到老生代中。

另一种情况，如果没有经历过 Scavenge 算法，会被复制到 To 空间，但是如果 To 空间内存占比已经超过 25%,那么依旧会被转移到老生代。

老生代 #

老生代中有大量的存活对象，且 gc 频率较低，因此上述空间换时间的算法不适用，老生代采用的是*Mark-Sweep(标记清除，简称 MS)和Mark-Compact(标记整理，简称 MC)*算法来进行 gc。

还记得最早的 ie 浏览器，一谈起垃圾回收，想到的是引用计数算法，原理很简单，就是检查对象是否还有其他引用指向它，如果没有，就会回收该对象。它的缺点也很明显，无法处理循环引用，会导致内存泄漏

MS 算法分为标记和清除两个阶段，具体的步骤如下：

  • gc 回收器构建一个跟列表，用于从根节点去寻找可以被访问到的变量，也就是 Window 全局对象。
  • 从所有根节点，遍历其可以访问到的字节点，并标记成活动的，根节点不能到达的地方就是垃圾。
  • gc 回收器释放所有非活动内存。

但是光 MS 是不够的，这样释放可能会导致内存空间出现不连续的状态，会有很多碎片，因此需要增加一种 MC 算法。

算法实现也很简单，步骤如下:

  1. 假设在老生代中有 A、B、C、D 四个对象

  2. 标记阶段把 A 和 C 标记为活动的

  3. 整理阶段，把活动对象往内存的一端移动

  4. 回收活动对象的左侧内存

到此，也就完成了一次老生代的 gc 过程。

全停顿 #

js 是单线程的，垃圾回收会阻碍主线程的执行，垃圾回收执行完毕后才会恢复执行主任务的逻辑，这种行为称为全停顿（stop-the-world）。在标记阶段也同样会阻碍主线程执行，如果把整个内存遍历一遍，可能会造成卡顿。

增量标记 #

为了降低老生代垃圾回收导致的卡顿，，V8 把一次 gc 的标记分成了很多子标记过程，同时让 js 主线程执行和子标记交替执行，直到 gc 标记完成。

但是新问题来了：

  • 每次子过程标记了活动对象，那么下一次标记过程从哪里开始呢？
  • 在子标记的间隙，如果对已经标记过的对象修改了引用怎么办呢？

三色标记 #

标记清除方法用的是 0 和 1 来标记，即白和黑，标记过的是黑色，表示是活动对象。在执行一次完整的 gc 标记之前，gc 回收器把虽有的数据置为白色，然后从一组根对象开始，将所有能访问到的数据标记为黑色，结束之后，标记为黑色的就是活动对象，白色的就是待清理对象。

但是在增量标记中，无法判断已经标记到哪里了，因此增加了一个颜色表示状态。

  • 白色 指未标记
  • 灰色 指自身被标记，但是引用对象（子对象）未被标记
  • 黑色 指自身和子对象都被标记

参考一张图：

并且三色标记后，我们可以根据是否有灰色节点，判断整个标记是否完成。当有灰色标记时，下次从灰色节点开始继续标记，如果没有灰色节点，直接进入清理阶段。

这样就解决了第一个问题，。

写屏障 #

举一个场景，有 A,B,C 三个对象依次引用，在第一次增量标记阶段全部标为了黑色，然后执行 js，在 js 中我们把对象 B 的指向由 C 改为 D。然后恢复下一次增加标记。

这时候 D 是初始的白色，按照上述三色标记的结论，已经没有灰色的对象，说明已经标记完毕了，接下来进入清理阶段，白色对象 D 会被回收，显然是不正确的。

因此，V8 采用写屏障的机制，即一旦有黑色对象引用白色对象，会强制把白色对象标记为灰色，从而保证下一次增量标记阶段可以正确标记。也就是对象 B 的指向从对象 C 改为对象 D 后，白色对象 D 会被强制改为灰色。

这样，也就解决了第二个问题。
