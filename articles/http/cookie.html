Cookie #

由来 #

http 是一个无状态的协议，也就是每次的 http 请求都是独立的，不需要保留任何状态信息，但有时候，尤其在一些接口需要鉴权的场景下，我们需要带上登录的身份，这个时候，就需要Cookie。

定义 #

Cookie 本质上就是浏览器里面存储的键值对。实际的例子是：当向同一个域名下的页面发送请求，都会携带相同的 cookie，服务器拿到 cookie 之后，就能解析客户端的状态。此外，cookie 也是由服务端通过Set-Cookie字段来写入给客户端。

用途 #

Cookie 常用的场景如下

  • 会话管理，记录用户的登录状态，免去多次认证和登录；
  • 个性化信息，常见的 google 会将搜索设置（语言、每页条数）保存在 cookie 中
  • 记录用户的行为，国外很多网站使用 cookie 记录用户的点击流和某个产品的操作率；

属性 #

  • 生存时间 #

    Cookie 也是有有效期，通过Expires和Max-Age两个属性设置 （1）Expires 即过期时间 是一个绝对时间 （2）Max-Age 即最大存活时间，是一个相对时间

    如果 Cookie 过期了，会被直接删除，不会发送给服务端

  • 作用范围 #

    作用范围指定了在哪些域名和路径下可以使用 cookie，通过Domain和path设置，如果没有设置，会被默认为当前请求页面对应值

  • 安全 #

    （1）如果带上了 secure 字段，那么只能通过 https 传输 cookie；

    （2）如果带上了 HttpOnly，那么只能通过 http 传输，且 cookie 不允许通过 js 脚本来访问，例如 document.cookie 无法获取。（用来防范 XSS 攻击）

    （3）此外，还有个 SameSite 属性，Strict 表示浏览器完全禁止第三方请求携带 Cookie、Lax 只能在 get 方法提交表单或者 a 标签发送 get 请求的情况下携带 Cookie、None 模式请求会自动携带 Cookie； None 模式是默认模式，但是 Chrome 浏览器不同的版本有着差异，相对比较严格

Cookie 窃取 #

大多数网站把 Cookie 当作用户的唯一标识，这时，黑客可以窃取用户的 Cookie 模拟用户的请求行为，对于服务器来说无法区分是客户还是黑客。

主要有如下几种手段：

  • 网络监听 #

我们常用的 fiddler 可以监听所有的 http 请求，同样，在一些公共非加密的网络环境，尤其是 wifi，攻击者就能拿到 http 会话中的 cookie，然后模拟用户操作。

解决方案就是直接改成 https 协议，通过 SSL 对 Cookie 进行加密；

  • XSS 攻击 #

js 可以通过 document.cookie 脚本来获取 cookie 值，因此跨站脚本（XSS）可以向任意服务器发送请求。

解决方案：httponly 属性可以避免脚本读取 cookie。

缺点 #

  1. 容量小，体积上限只有 4kb，只能存储少量信息。
  2. 性能缺陷，如果不指定 domain 和 path，域名下的任一地址都会携带 cookie，不管需不需要 cookie，浪费性能
  3. 安全缺陷，如上述几种攻击；
